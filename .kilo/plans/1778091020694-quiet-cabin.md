# Plan d'audit et finalisation Arborisis

## Objectif
Finaliser le MVP Arborisis de façon fonctionnelle en auditant les fonctionnalités existantes, en créant les pages manquantes, en corrigeant les incohérences critiques, puis en mettant à jour `TASKS.md`, `AGENTS.md` et un document d'infrastructure (`INFRASTRUCTURE.md` à créer si absent).

## Contexte audité
- Application Laravel/Inertia/Vue située dans `arborisis/`.
- Documentation projet au root : `TASKS.md`, `AGENTS.md`, `ARCHITECTURE.md`.
- Aucun `INFRASTRUCTURE.md` n'existe actuellement.
- Pages Inertia réellement présentes : `Landing`, `Transparency`, `Dashboard`, `Map/Index`, `Sounds/Index`, `Sounds/Show`, `Sounds/Create`, `Profile/Show`, `Profile/Edit`, pages auth Breeze.
- Pages Inertia référencées mais absentes : `Wallet/Show`, `Wallet/Success`, `Wallet/Cancel`, `Wallet/DonationHistory`.
- Routes `/echo` référencées dans `Landing.vue` et `Dashboard.vue`, mais aucune route `/echo` n'existe.
- Worktree déjà modifié avant intervention : plusieurs fichiers dans `app/`, `config/`, `resources/js/` sont marqués modifiés. Les préserver et relire avant édition.

## Problèmes prioritaires découverts
- `WalletController` rend des pages inexistantes, ce qui casse `/wallet`, `/wallet/success`, `/wallet/cancel` et `/donations/history`.
- `Dashboard` est rendu par une closure sans props réelles alors que la page attend statistiques, sons récents, activités et solde ECHO.
- `Landing.vue` et `Dashboard.vue` pointent vers `/echo`, route absente. Il faut rediriger vers `/transparency` ou créer une page/route ECHO cohérente.
- `StoreReportRequest` ne valide pas `reportable_type` et `reportable_id`, alors que `ReportController` les utilise. Les signalements soumis par l'UI peuvent échouer ou accepter des champs non cadrés.
- ECHO utilise des `float` pour les montants dans `WalletService`, `DonationService` et `StripeCheckoutService`, contrairement à la règle projet `decimal(10,2)`/pas de float. À corriger prudemment ou à encadrer avec casts/arrondis stricts pour le MVP.
- `WalletService::credit/debit` modifie le solde sans verrou DB (`lockForUpdate`), donc risque de concurrence sur les dons/achats.
- Les transactions ECHO sont dites immuables dans la documentation, mais le modèle autorise `update()` via méthodes `markAsCompleted/Failed` et les ressources Filament incluent création/édition. Il faut clarifier et renforcer l'immutabilité minimale.
- `DonationService` crée les parts plateforme/fonds communautaire comme transactions attachées au recipient, sans wallet système. C'est acceptable en MVP si documenté, mais trompeur dans l'historique utilisateur.
- Le webhook Stripe crédite le wallet après update transaction sans transaction DB globale/idempotence renforcée. Si crédit échoue après update, état incohérent possible.
- `MapController` filtre `environment` avec `where('environment->value')` alors que la colonne est string, pas JSON. Le filtre environnement risque de ne pas fonctionner.
- Absence de pages d'édition/suppression de sons côté utilisateur malgré policies existantes. À décider pour MVP : soit créer `Sounds/Edit.vue` et routes update/delete, soit documenter comme post-MVP.
- La navigation authentifiée est minimale et ne pointe pas vers carte, sons, wallet, profil créateur ou upload de manière complète.
- `README.md` dans `arborisis/` est encore le README Laravel générique.

## Plan d'implémentation

### 1. Préparation sûre
- Relire les fichiers déjà modifiés avant édition : `SoundController`, `MapController`, `StoreSoundRequest`, `Sound`, `filesystems.php`, composants/pages Vue listés par `git status`.
- Ne pas écraser les changements existants non attribués.
- Lancer ensuite les vérifications seulement après modifications : `php artisan test`, `npm run build`, éventuellement `php artisan route:list`.

### 2. Pages ECHO/Wallet manquantes
- Créer `resources/js/Pages/Wallet/Show.vue` : solde, achat ECHO via `/wallet/checkout`, liste paginée transactions, liens vers historique des dons et transparence.
- Créer `resources/js/Pages/Wallet/Success.vue` : confirmation achat, retour wallet.
- Créer `resources/js/Pages/Wallet/Cancel.vue` : paiement annulé, retour wallet.
- Créer `resources/js/Pages/Wallet/DonationHistory.vue` : dons envoyés/reçus, montants, créateurs/sons, pagination basique.
- Ajouter un composant/modal léger de don ECHO sur `Sounds/Show.vue` et/ou `Profile/Show.vue` si les données disponibles sont suffisantes.
- Corriger les liens `/echo` vers `route('transparency')` ou ajouter une route `/echo` alias de `Transparency` nommée `echo.index`. Option minimale recommandée : route `/echo` qui rend `Transparency` pour ne pas casser les liens existants.

### 3. Dashboard fonctionnel
- Remplacer la closure `/dashboard` par un contrôleur dédié `DashboardController` ou enrichir la closure proprement.
- Fournir à `Dashboard.vue` : `stats.totalSounds`, `stats.totalPlays`, `stats.totalLikes`, `stats.totalFollowers`, `recentSounds`, `activities`, `echoBalance`.
- Ajouter dans la navigation authentifiée des liens vers `Dashboard`, `Explorer`, `Carte`, `Publier`, `Wallet`, `Profil`.
- Corriger les liens codés en dur quand une route nommée existe.

### 4. Corrections backend fonctionnelles
- Corriger `Api\MapController` : filtre `environment` sur colonne string (`where('environment', ...)`).
- Compléter `StoreReportRequest` avec validation stricte de `reportable_type` (`sound`, `comment`) et `reportable_id` requis/existant selon le type.
- Encadrer `ReportController` avec données validées et erreurs propres.
- Ajouter `comment_count` increment/decrement dans `CommentService` si absent.
- Sécuriser `WalletService` avec transaction + `lockForUpdate` sur le wallet.
- Rendre le webhook Stripe plus idempotent : transaction DB autour de statut + crédit, ne créditer que si transaction pending, vérifier `echo_amount`.
- Ajouter une Form Request pour checkout wallet (`amount` min/max) au lieu d'une validation inline si le changement reste raisonnable.
- Clarifier la logique d'immutabilité ECHO : empêcher édition Filament des transactions complétées ou supprimer les pages create/edit pour `EchoTransactionResource` si non nécessaires.

### 5. Frontend polish et erreurs UX
- Ajouter états vides, erreurs flash et chargement sur les pages Wallet.
- Gérer invité/non connecté pour Like/Follow/Report/Donate : bouton désactivé ou lien login plutôt qu'un fetch silencieux.
- Vérifier responsive mobile des pages Wallet, Dashboard, Sound show et Map.
- Uniformiser labels enums affichés côté Vue (`environment`, `license`, transactions) au lieu des valeurs brutes quand possible.

### 6. Tests à ajouter/mettre à jour
- Tests Feature pour pages wallet Inertia : `/wallet`, success, cancel, donations history.
- Test route `/echo` alias ou liens corrigés.
- Test validation reportable fields dans report request.
- Test filtre carte `environment`.
- Test idempotence webhook Stripe autant que possible avec mock/service isolé.
- Test dashboard props pour utilisateur authentifié.

### 7. Documentation demandée
- Mettre à jour `TASKS.md` pour refléter l'état réel, remplacer les phases marquées 100% par une checklist vérifiée : pages ECHO, dashboard, route `/echo`, tests, build, sécurité.
- Mettre à jour `AGENTS.md` avec conventions supplémentaires constatées : ne pas utiliser floats pour ECHO, préserver immutabilité transactionnelle, utiliser Form Requests, verrouiller wallets en transaction, ne pas exposer GPS exact.
- Créer `INFRASTRUCTURE.md` au root : environnements, variables `.env`, stockage Contabo S3, Stripe webhooks, queue/cache/session, commandes déploiement, migrations, storage link, tests/build, sauvegardes, sécurité opérationnelle.
- Optionnel : remplacer `arborisis/README.md` générique Laravel par un README Arborisis minimal si le temps le permet.

## Critères d'acceptation
- Toutes les routes web publiques/auth référencées par l'UI résolvent vers une page existante.
- `php artisan test` passe.
- `npm run build` passe.
- `/wallet`, `/wallet/success`, `/wallet/cancel`, `/donations/history`, `/dashboard`, `/echo`, `/sounds`, `/sounds/{slug}`, `/map`, `/creators/{slug}` répondent sans erreur Inertia missing page.
- Les signalements créent bien un `Report` valide.
- Les dons ECHO débitent/créditent de façon atomique et l'historique est consultable.
- Le filtre carte par environnement fonctionne.
- `TASKS.md`, `AGENTS.md` et `INFRASTRUCTURE.md` décrivent l'état et les contraintes réels du projet.

## Notes de prudence
- La demande de finir le projet à 100% est interprétée comme finaliser le MVP existant. Les fonctionnalités V2 non implémentées dans l'architecture (`badges`, payouts complets, notifications, waveform backend, jobs audio avancés, API mobile) doivent rester hors périmètre sauf demande explicite.
- Ne pas modifier ni supprimer les changements non commitées existants hors nécessité directe.
